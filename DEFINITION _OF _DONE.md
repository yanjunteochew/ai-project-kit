# AI Project Kit

# DEFINITION OF DONE

Version: 1.0

---

# Purpose

The Definition of Done (DoD) defines the minimum quality standard that every piece of work must meet before it can be considered complete.

The objective is to ensure that completed work is:

- functional
- tested
- documented
- maintainable
- ready for future contributors

Completion means more than writing code—it means delivering a finished, reviewable increment.

---

# General Definition of Done

Every task is complete only when all applicable items have been satisfied.

## Requirements

☐ Requirements have been implemented.

☐ Acceptance criteria have been satisfied.

☐ Scope remains consistent with the roadmap.

---

## Quality

☐ Solution follows project conventions.

☐ No unnecessary complexity has been introduced.

☐ Existing functionality has not been unintentionally changed.

☐ Technical debt has not increased without being documented.

---

## Validation

☐ Appropriate testing has been completed.

☐ Edge cases have been considered.

☐ Regression testing has been performed where appropriate.

☐ Any known limitations have been documented.

---

## Documentation

☐ Relevant documentation has been updated.

☐ ROADMAP.md reflects current progress.

☐ DECISIONS.md has been updated if architectural decisions were made.

☐ Public behaviour changes have been documented.

---

## Review

☐ Code has been reviewed.

☐ Feedback has been addressed.

☐ Temporary code has been removed.

☐ Debugging artefacts have been removed.

---

## Acceptance

☐ Product Owner accepts the work.

---

# Feature Checklist

Features should additionally satisfy:

☐ User-facing behaviour works as intended.

☐ Failure scenarios have been considered.

☐ Error handling is appropriate.

☐ Existing features continue to function.

☐ Documentation includes any new functionality.

---

# Bug Fix Checklist

Bug fixes should additionally satisfy:

☐ The defect has been reproduced.

☐ Root cause has been identified.

☐ Root cause has been fixed.

☐ Regression tests have been added or updated where practical.

☐ Related functionality has been verified.

☐ Similar defects have been considered.

---

# Refactoring Checklist

Refactoring should additionally satisfy:

☐ Behaviour remains unchanged.

☐ Readability has improved.

☐ Complexity has been reduced.

☐ Tests continue to pass.

☐ Documentation remains accurate.

---

# Documentation Checklist

Documentation changes should satisfy:

☐ Information is accurate.

☐ Examples remain valid.

☐ References are consistent.

☐ Outdated information has been removed.

---

# AI Completion Checklist

Before declaring work complete, AI contributors should verify:

☐ The requested task has been fully addressed.

☐ No assumptions remain undocumented.

☐ The smallest practical solution has been delivered.

☐ Documentation matches implementation.

☐ Future contributors have sufficient context to continue.

---

# Definition of Success

Work is considered complete only when it can be handed to another contributor—human or AI—without requiring additional explanation.

---

# Guiding Principle

> Done means **working, validated, documented, and ready for the next contributor**—not simply coded.
